Model Deployment Explanation:

  • Local Development (PC using Git):

    • Developed and tested the code locally using Git for version control.

  • GitHub Repository:

    • Pushed the code to a GitHub repository where the configuration files and codebase reside.

  • GitHub Actions Workflows:

    • Defined workflows in the GitHub repository using GitHub Actions.
    • These workflows are triggered by events such as pushes to the repository.

  • GitHub Actions Runner (Self-hosted):

    • Utilized a self-hosted GitHub Actions runner.
    • This runner is configured on my AWS EC2 instance.

  • AWS EC2 Instance:

    • Configured an EC2 instance to act as a runner for GitHub Actions.
    • This EC2 instance serves as a bridge between my GitHub repository and AWS services.

  • GitHub Actions Workflow Execution:

    • When triggered, GitHub Actions fetches the workflow defined in the repository.
    • The workflow specifies tasks such as building Docker images, running tests, or deploying the application.

  • Docker Image:

    • Built a Docker image containing my application and dependencies.
    • This image is used for deployment and can be stored in a Docker registry.

  • ECR (Elastic Container Registry):

    • Stored the Docker images in Amazon ECR.
    • ECR provides a secure and scalable repository for storing Docker images.

  • Deployment to EC2:

    • Configured my GitHub Actions workflow to deploy the Docker image to my EC2 instance.
    • This deployment process involves pulling the Docker image from ECR and running it on the EC2 instance.

  • Final Deployment on EC2:

    • My application is now deployed and running on the EC2 instance.

This workflow enables to automate the deployment of the application from GitHub repository to an AWS EC2 instance via Docker images stored in ECR, using GitHub Actions as the orchestrator and a self-hosted runner on an EC2 instance.

Configuration Work Flow Explanation

Configuration Files:

  • config.yaml, schema.yaml, and params.yaml contain configuration settings, dataset schema, and model parameters, respectively.
  • These files define various aspects of the machine learning pipeline such as data ingestion, transformation, and model training.

Configuration Manager:

  • The Configuration Manager in src/configuration reads these configuration files and provides configuration objects to other components.
  • It acts as a central hub for accessing configuration settings throughout the pipeline.

Entity Definition Workflow/ Entity Modules:

  • src/entity contains modules defining data structures used within the pipeline, such as configuration entities and data validation entities.
  • These modules define classes representing different entities like DataIngestionConfig, DataValidationConfig, etc.

Component Implementation Workflow/ Component Modules:

  • src/components contains modules implementing specific functionalities of the machine learning pipeline, such as data ingestion, validation, transformation, model training, and evaluation.
  • Each module contains a class representing a specific component, such as DataIngestion, DataValidation, ModelTrainer, etc.

Constants Definition Workflow/ Constants Module:

  • src/constants contains modules defining constants used across the pipeline.
  • These constants may include file paths, default parameters, or any other values that are reused throughout the codebase.

Pipeline Orchestration Workflow/ Pipeline Modules:

  • src/pipeline contains modules orchestrating the execution of the machine learning pipeline stages.
  • Each module represents a specific stage of the pipeline, such as data ingestion, validation, transformation, model training, and evaluation.
  • These modules import and utilize components from src/components to execute the corresponding tasks for each stage.

Main Execution Workflow/ Main Script:

  • main.py orchestrates the execution of different stages of the machine learning pipeline in sequence.
  • It imports and runs the various training pipelines from src/pipeline to execute each stage.

Web Application Workflow/ Flask Application:

  • app.py implements a Flask web application that provides endpoints for triggering the training process (/train) and making predictions (/predict).
  • It utilizes components and pipelines from src directory to perform training and prediction tasks.

This end-to-end workflow analysis demonstrates how each file and directory in the provided structure contributes to different aspects of the machine learning pipeline, from configuration setup to pipeline execution and web interface integration.
